事件驱动架构 ​

定义 ​

组件之间通过事件进行通信，而非直接调用。

在 nanobot 中的应用 ​

用户发送消息
    │
    ▼ 事件
TelegramChannel 收到消息
    │
    ▼ 发布事件
MessageBus.inbound
    │
    ▼ 消费事件
AgentLoop 处理消息
    │
    ▼ 发布事件
MessageBus.outbound
    │
    ▼ 消费事件
TelegramChannel 发送响应

好处 ​

  • ✅ 解耦：组件互不依赖
  • ✅ 异步：不阻塞等待
  • ✅ 可扩展：易于添加新组件

实现位置 ​

  • 文件: nanobot/bus/queue.py
  • 类: MessageBus
  • 事件: InboundMessage, OutboundMessage

消息总线模式 ​

定义 ​

使用中央消息总线解耦组件，实现异步通信。

架构图 ​

┌─────────────────────────────────────────────────────────────┐
│                      MessageBus                             │
│  ┌─────────────────────┐      ┌─────────────────────┐     │
│  │   inbound Queue     │      │  outbound Queue     │     │
│  │  (生产者 → 消费者)   │      │  (消费者 → 生产者)   │     │
│  └─────────────────────┘      └─────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
         ▲                              │
         │                              │
    publish_inbound              publish_outbound
         │                              ▲
         │                              │
┌────────────┐                  ┌────────────┐
│ Channels   │                  │ AgentLoop  │
│ (生产者)    │                  │ (消费者/生产者)│
└────────────┘                  └────────────┘

为什么选择 ​

  1. 解耦 Channels 和 Agent

    • Channel 不需要知道 Agent 的存在
    • Agent 不需要知道 Channel 的实现

  2. 异步处理

    • 使用 asyncio.Queue 实现非阻塞通信
    • 支持并发处理多个消息

  3. 缓冲机制

    • 消息不会丢失
    • 平滑流量峰值

代码示例 ​

python
# 生产者
class TelegramChannel:
    async def on_message(self, msg):
        inbound_msg = InboundMessage(...)
        await self.bus.publish_inbound(inbound_msg)

# 消费者
class AgentLoop:
    async def run(self):
        while True:
            msg = await self.bus.consume_inbound()
            await self._process(msg)

插件式架构 ​

定义 ​

核心功能提供扩展点，插件可以在这些点插入功能。

在 nanobot 中的应用 ​

扩展点插件类型位置
Channels聊天平台nanobot/channels/
ProvidersLLM 提供商nanobot/providers/
Tools工具nanobot/agent/tools/
Skills技能nanobot/skills/

插件接口 ​

python
# Channel 插件
class BaseChannel(ABC):
    @abstractmethod
    async def start(self): pass
    @abstractmethod
    async def stop(self): pass
    @abstractmethod
    async def send(self, msg): pass

# Provider 插件
class LLMProvider(ABC):
    @abstractmethod
    async def chat(self, messages, tools, **kwargs):
        pass

# Tool 插件
class Tool(ABC):
    @property
    @abstractmethod
    def name(self) -> str: pass
    @abstractmethod
    async def execute(self, **kwargs) -> str:
        pass

注册机制 ​

python
# Channel 注册
channel = TelegramChannel(config, bus)
channel_manager.register(channel)

# Tool 注册
tool = ReadFileTool(workspace)
tool_registry.register(tool)

# Provider 注册
PROVIDERS["anthropic"] = ProviderSpec(...)

好处 ​

  • ✅ 开放性：用户可以扩展
  • ✅ 模块化：功能独立
  • ✅ 可维护：易于更新

分层架构 ​

定义 ​

将系统分为多个层次，每层只与相邻层通信。

nanobot 的分层 ​

┌─────────────────────────────────────────────────────────────┐
│  第 1 层：聊天平台层                                        │
│  Telegram, Discord, Feishu, ...                             │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  第 2 层：通道适配层                                        │
│  BaseChannel, ChannelManager                               │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  第 3 层：消息总线层                                        │
│  MessageBus, InboundMessage, OutboundMessage               │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  第 4 层：代理核心层                                        │
│  AgentLoop, ContextBuilder, SessionManager, ToolRegistry   │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  第 5 层：外部服务层                                        │
│  LLM Providers, Tools, Cron, Heartbeat                     │
└─────────────────────────────────────────────────────────────┘

层次职责 ​

层次职责不应做
聊天平台与用户交互处理业务逻辑
通道适配格式转换直接调用 LLM
消息总线消息传递处理消息内容
代理核心业务逻辑处理平台细节
外部服务具体功能管理会话状态

好处 ​

  • ✅ 关注点分离：每层专注于自己的职责
  • ✅ 易于理解：层次清晰
  • ✅ 灵活替换：可以替换某一层的实现

架构原则 ​

1. 单向依赖 ​

上层可以依赖下层，下层不应依赖上层。

✅ 正确：
AgentLoop → MessageBus → Channel

❌ 错误：
Channel → AgentLoop

2. 接口隔离 ​

组件通过接口通信，而非具体实现。

python
# ✅ 使用抽象
def __init__(self, provider: LLMProvider):
    self.provider = provider

# ❌ 使用具体类
def __init__(self, provider: AnthropicProvider):
    self.provider = provider

3. 依赖注入 ​

通过构造函数注入依赖。

python
# ✅ 依赖注入
class AgentLoop:
    def __init__(
        self,
        bus: MessageBus,
        provider: LLMProvider,
        session_manager: SessionManager,
    ):
        self.bus = bus
        self.provider = provider
        self.sessions = session_manager
