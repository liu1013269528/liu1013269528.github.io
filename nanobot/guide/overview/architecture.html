核心设计理念 ​

1. 解耦 (Decoupling) ​

原则：各组件之间通过消息总线通信，互不依赖。

┌────────────┐      消息      ┌────────────┐
│  Channel   │ ═════════════▶ │   Agent    │
│            │ ◀═════════════ │            │
└────────────┘      消息      └────────────┘
     不依赖                            不依赖
      具体实现                          具体渠道

好处：

  • ✅ 添加新渠道无需修改 Agent 代码
  • ✅ Agent 可以独立测试
  • ✅ 各组件可以独立演进

2. 异步优先 (Async-First) ​

原则：全面使用 asyncio 实现高并发。

python
# 示例：消息处理是异步的
async def run(self):
    while True:
        msg = await self.bus.consume_inbound()  # 异步等待
        await self._dispatch(msg)  # 异步处理

好处：

  • ✅ 同时处理多个用户的请求
  • ✅ I/O 操作不阻塞（网络、文件）
  • ✅ 充分利用系统资源

3. 插件化 (Plugin-Based) ​

原则：Channels、Providers、Tools 都是可插拔的。

                    ┌──────────────┐
                    │  Registry    │
                    │  (注册表)     │
                    └──────┬───────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
      ┌────▼────┐    ┌────▼────┐    ┌────▼────┐
      │Plugin A │    │Plugin B │    │Plugin C │
      └─────────┘    └─────────┘    └─────────┘
      随时添加/移除

好处：

  • ✅ 用户可以自定义扩展
  • ✅ 社区可以贡献插件
  • ✅ 核心代码保持简洁

4. 轻量设计 (Lightweight) ​

原则：最小化依赖，保持代码简洁。

对比项nanobot其他框架
代码量~4000 行数万~数十万行
核心依赖5-10 个数十个
启动时间秒级分钟级
内存占用< 100MB数百 MB~GB

好处：

  • ✅ 易于阅读和理解
  • ✅ 快速启动和部署
  • ✅ 低资源消耗

各层职责 ​

第 1 层：聊天平台层 ​

职责：与用户直接交互的平台（Telegram、Discord 等）

nanobot 不关心：

  • 具体是哪个平台
  • 平台的特性差异
  • 消息格式差异

通道适配层负责转换：

平台特有消息 → 统一的 InboundMessage
               ↓
          Agent 处理
               ↓
统一的 OutboundMessage → 平台特有消息

第 2 层：通道适配层 ​

职责：将不同平台的差异统一化

核心组件：

  • BaseChannel：抽象基类，定义统一接口
  • ChannelManager：管理所有通道实例

设计模式：

  • 适配器模式：将不同平台接口适配为统一接口
  • 工厂模式：根据配置创建对应的通道实例
python
class BaseChannel(ABC):
    @abstractmethod
    async def start(self): pass

    @abstractmethod
    async def stop(self): pass

    @abstractmethod
    async def send(self, msg): pass

第 3 层：消息总线层 ​

职责：解耦通道和代理，提供异步消息传递

核心组件：

  • MessageBus：消息队列容器
  • InboundMessage：入站消息格式
  • OutboundMessage：出站消息格式

设计模式：

  • 消息总线模式：使用队列解耦组件
python
class MessageBus:
    def __init__(self):
        self.inbound = asyncio.Queue()   # 入站
        self.outbound = asyncio.Queue()  # 出站

第 4 层：代理核心层 ​

职责：消息处理的核心逻辑

核心组件：

组件职责
AgentLoop主循环，消息分发和迭代
ContextBuilder构建 LLM 上下文
SessionManager管理对话历史
MemoryStore长期记忆管理
SkillsLoader技能文件加载
SubagentManager后台任务管理
ToolRegistry工具注册表

设计模式：

  • 事件循环模式：持续处理消息事件
  • 构建器模式：ContextBuilder 分步骤构建上下文
  • 命令模式：工具封装为可执行命令

第 5 层：外部服务层 ​

职责：提供具体功能实现

服务说明
LLM Providers提供大模型调用
Tools提供可执行的操作
Cron Service定时任务调度
Heartbeat Service定期触发任务

技术选型 ​

编程语言：Python 3.11+ ​

为什么选择 Python？

优势说明
异步支持asyncio 原生支持，性能优秀
AI 生态与 LLM SDK 集成良好
易读性代码简洁，适合学习
跨平台可在 Windows/macOS/Linux 运行

核心依赖 ​

库用途为什么选择
asyncio异步编程Python 内置，性能优秀
pydantic数据验证类型安全，自动校验
typerCLI 框架简洁易用，自动生成帮助
loguru日志记录功能强大，使用简单
litellmLLM 统一接口支持 100+ 模型

架构模式 ​

模式应用位置理由
消息总线MessageBus解耦通道和代理
事件驱动AgentLoop响应消息事件
插件式Channels/Providers/Tools可扩展架构
注册表ToolRegistry动态发现和管理
适配器Channel 实现统一不同平台接口
构建器ContextBuilder分步骤构建复杂对象
命令Tool封装操作为对象

架构优势 ​

1. 易于理解 ​

  • 清晰的分层结构
  • 每层职责明确
  • 代码简洁易读

2. 易于扩展 ​

  • 添加新平台：继承 BaseChannel
  • 添加新工具：继承 BaseTool
  • 添加新 LLM：扩展 ProviderRegistry

3. 易于测试 ​

  • 各组件独立，可单独测试
  • 依赖注入，方便 Mock
  • 异步设计，易于模拟

4. 易于部署 ​

  • 轻量级，资源消耗小
  • 无需复杂配置
  • 支持多实例运行
